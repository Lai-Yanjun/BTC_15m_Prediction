# 树莓派代理出站与下单连通性一键排障（Shadow 版）

适用场景：
- 不在代码内建 VPN 隧道，只依赖本机代理（通常 Clash）。
- 通过进程环境变量让所有交易/API 请求统一走代理出口。
- 先验证连通性闭环，再启动 shadow 长跑。

---

## 0) 进入项目目录

```bash
cd ~/apps/predicta-live
```

---

## 1) 方案总览（核心原则）

1. **代理注入点在启动层**：`run_shadow.py` 与 `check_order_connectivity.py` 支持  
   `--proxy/--http-proxy/--https-proxy/--no-proxy`，启动时写入：
   - `http_proxy` / `https_proxy`
   - `HTTP_PROXY` / `HTTPS_PROXY`
   - `no_proxy` / `NO_PROXY`
2. **请求层自动走代理**：后续 `httpx` / `requests` / `py_clob_client` 对  
   `data-api.polymarket.com`、`clob.polymarket.com` 的请求，依赖这些环境变量出站。
3. **验证闭环再开跑**：必须先跑 `check_order_connectivity.py` 看 `egress_ip` 与下单测试结果。

---

## 2) 检查 Clash 端口是否在监听

```bash
ss -lntp | rg "7890|7891|7897"
```

常见端口（以本机配置为准）：
- `7890`：HTTP 代理
- `7891/7897`：SOCKS 代理

建议直接用 HTTP 代理地址，例如：`http://127.0.0.1:7897`。

---

## 3) 连通性验证（必须先做）

先只测出口与 API：

```bash
./.venv/bin/python check_order_connectivity.py \
  --config trade_config.yaml \
  --proxy http://127.0.0.1:7897
```

再做真实小额下单 + 撤单闭环：

```bash
./.venv/bin/python check_order_connectivity.py \
  --config trade_config.yaml \
  --proxy http://127.0.0.1:7897 \
  --test-order \
  --order-usdc 1.0 \
  --cancel-after-sec 2
```

成功标准：
- 输出中有 `egress_ip`（当前实际出口 IP）
- `data_api.ok=true`、`clob.ok=true`
- 开启 `--test-order` 时，`order_test.ok=true`

若结果为 `GEO_BLOCK_403`：表示当前出口仍被地区风控拦截（代理未生效或发生泄漏）。

---

## 4) 启动 Shadow（前台）

```bash
./.venv/bin/python run_shadow.py \
  --config trade_config.yaml \
  --proxy http://127.0.0.1:7897
```

单轮冒烟：

```bash
./.venv/bin/python run_shadow.py \
  --config trade_config.yaml \
  --run-once \
  --proxy http://127.0.0.1:7897
```

---

## 5) systemd 后台关键点（避免“前台可用、后台 403”）

仓库提供模板：`deploy/pm_shadow.service`。

要点：
- `ExecStart` 建议直接带 `--proxy http://127.0.0.1:7897`
- 模板里 `Environment=` 代理项默认注释
- 若你不在 `ExecStart` 传 `--proxy`，必须取消注释 `Environment=` 代理变量

部署示例：

```bash
sudo cp deploy/pm_shadow.service /etc/systemd/system/pm_shadow.service
sudo systemctl daemon-reload
sudo systemctl enable --now pm_shadow
sudo systemctl status pm_shadow
journalctl -u pm_shadow -f
```

---

## 6) 常见失败根因（按命中率排序）

### A. Clash 端口变化

症状：昨天可用，今天全 403/超时。  
处理：重查监听端口，统一改脚本参数与 systemd 配置。

### B. 后台进程没注入代理变量

症状：前台脚本可用，`systemd` 后台报 403。  
处理：确保 `ExecStart` 带 `--proxy`，或在 service 里取消注释 `Environment=`。

### C. IPv6 泄漏 / TUN 非全局

症状：`egress_ip` 不稳定或与预期出口不一致。  
处理：在 Clash 端启用全局/TUN 全局接管，必要时禁用 IPv6 直连泄漏路径。

### D. `GEO_BLOCK_403`

含义：请求已到目标服务，但出口被地区风控。  
处理：先看 `check_order_connectivity.py` 输出的 `egress_ip`，再调整节点与路由策略。

---

## 7) 一条命令快速体检（可直接复制）

```bash
cd ~/apps/predicta-live && \
./.venv/bin/python check_order_connectivity.py --config trade_config.yaml --proxy http://127.0.0.1:7897 --test-order --order-usdc 1.0 --cancel-after-sec 2 && \
./.venv/bin/python run_shadow.py --config trade_config.yaml --proxy http://127.0.0.1:7897
```
